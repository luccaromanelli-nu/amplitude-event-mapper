# Outputs

Every output lands in `projects/<project_name>/`. Nothing else is written.

## Files

### `source-docs.md`

Inventory of where every input came from. Generated by gather agents. Contains:

- `## Figma sources` — table of Figma URLs walked, frames per cell, interactions found
- `## Miro sources` — boards read, cards extracted
- `## Confluence sources` — pages read, events extracted
- `## Amplitude sources` — Databricks query target, country, time window, row count

Use this file to audit "where did this event come from?" later.

### `existing-events.md`

Normalized union of every event mentioned by every source. Sectioned by source so you can see provenance. Contains:

- `## Figma screen inventory` — every node from Figma
- `## Miro documented events` — every event card from Miro
- `## Confluence documented events` — every event from Confluence
- `## Amplitude (production)` — every event firing in production

### `cross-check.md`

Gap matrix. Sections:

- Stale or broken
- Undocumented
- Property mismatches
- Uncovered Figma nodes
- A/B coverage gap (when ab_test=true)
- Summary counts

Empty sections render with `_(none)_` so reviewers can confirm the section was checked.

### `proposed-events.md`

The deliverable engineers review. Sections:

- Project header (name, country, experiment, scope, generated date, status)
- Summary (2-3 sentences)
- Out of scope (explicit)
- Proposed events (one block per event, or comparison table when cells share an event)
- Open questions (numbered checklist of every `[CONFIRM]` and `[NEEDS-HUMAN]`)
- Approval instructions

Tag conventions:

- `[NEW]` — value not in production yet
- `[CONFIRM]` — value exists somewhere but needs engineer validation
- `[NEEDS-HUMAN]` — edge case requiring manual decision

### `catalog-suggestions.md`

Optional. Lists events from the proposal worth promoting to the shared catalog. Read-only against `catalog/events.md`. Humans decide separately whether to apply.

### `final-event-map.md`

Generated only after `APPROVED` marker is created. Same shape as `proposed-events.md` but with `[CONFIRM]` tags stripped, open-questions checklist removed, and a finalized header added. `[NEW]` tags retained so engineers know what is introduced for the first time.

## Lifecycle

```
input.yaml
   │
   ↓ /propose
source-docs.md, existing-events.md
   │
   ↓ (still in /propose)
cross-check.md
   │
   ↓ (still in /propose)
proposed-events.md, catalog-suggestions.md
   │
   ↓ human review, [CONFIRM] resolved, touch APPROVED
   ↓ /propose (re-run) or /finalize
final-event-map.md
```

## What is NOT produced

- No code (engineers may use a separate skill to generate code from `final-event-map.md`).
- No Slack messages.
- No Jira tickets.
- No Confluence pages.
- No Google Docs / Sheets / Slides.
- No git commits.

If you ever see one of these, the side-effect guard has been bypassed — file a bug.
